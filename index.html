01 — The problem

How well is your Claude environment actually performing?

The Claude ecosystem has a layered architecture. How content is distributed across those layers determines whether Claude performs at its potential or works against itself. The hard part isn't building a Project or wiring a Claude Code repo — it's seeing what's quietly working against you.

Architectural inefficiencies don't announce themselves. They surface as vague symptoms: output that drifts, instructions that get ignored intermittently, quality that degrades over long conversations, autonomous agents that escalate work they shouldn't or barrel through halts they should respect. These Skills surface those structural issues, triage them by impact, and produce targeted fixes.

They treat the entire ecosystem as a unified system — scoring each layer, detecting cross-layer conflicts, diagnosing behavioral tendencies, gating handoffs to autonomous execution, and building Projects and Claude Code environments that are structurally sound from the start.

02 — The architecture

Two surfaces. Four scopes.

Two surfaces, layered architecture across each. Skills are mapped to the layers they impact — a Skill that builds knowledge files appears on the Knowledge Files layer because that's the layer it modifies; a Skill that audits five layers at once appears on each of them.

One Skill spans Global and Project scopes. rootnode-full-stack-audit runs Project audit + Global audit + Cross-Layer Alignment in a single pass — the only Skill in the catalog that audits two scopes together rather than within one.

06 — The workflow

End-to-end optimization.

root.node covers four architectural scopes, each with its own layers. The Skills map to how the work flows. Design a Claude Project, build it out, and hand off to Claude Code when execution warrants autonomy. Design the repo, deploy, and iterate. What you learn during shipping enhances the next design pass.

root.node optimizes every step of the loop. Diagnose what's structurally working against you, build with tested components, and correct when the architecture drifts.

User Preferences

Communication style, expertise level, behavioral defaults. Everything here persists across all Projects and all conversations. The foundation layer: what Claude knows about you before any Project loads.

Preferences are too vague to guide Claude's behavior. Redundant rules get restated in every Project. Stable patterns stay trapped in Project Memory instead of being promoted globally.

Evaluates Preference precision, detects redundancies with Project-scoped instructions, and identifies patterns ready for promotion from Project Memory.

Styles

Tone, format, and delivery patterns. Applied by selection and can be switched per conversation. Controls how Claude communicates, separate from what it knows or how it reasons.

Style settings conflict with formatting rules hardcoded in Custom Instructions. Tone preferences get scattered across Projects instead of centralized.

Detects tension between Style settings and Custom Instructions formatting. Identifies when delivery preferences belong in a Style rather than being hardcoded per Project.

Global Memory

Cross-project context and learned preferences. Auto-synthesized from conversations. Orientation facts that help Claude understand who you are and what matters to you, loaded into every conversation.

Memory accumulates stale facts, redundant entries, and reference-depth content that belongs in knowledge files. Bloated Memory consumes context budget across every conversation.

Audits for redundancy and staleness. Rebalances content between Memory, Preferences, and knowledge files. Tracks Memory's contribution to context overhead.

Skills

Procedural knowledge packaged in SKILL.md files. Activated automatically by task relevance. Skills use progressive disclosure: a lightweight description loads at startup, the full methodology loads on demand.

Installed Skills collide with Project knowledge files covering the same territory. The Skill portfolio has gaps that force manual workarounds. Overhead from unused Skills consumes context budget.

Evaluates Skill portfolio fitness, detects collisions with knowledge files, and converts design specs into deployment-ready Skill packages.

MCP Connectors

External service integrations like Gmail, Slack, Google Drive, and Notion. Tools loaded on demand through deferred loading: configured once, available everywhere.

Projects assume connector capabilities that aren't configured. Connector overhead goes untracked. Alignment between available connectors and Project needs drifts as Projects evolve.

Checks connector alignment with Project needs, detects assumed capabilities without configured connectors, and accounts for connector overhead.

Project Memory

Phase awareness, active constraints, key decisions. Auto-synthesized per Project. The orientation layer that helps Claude pick up where you left off: what you're working on, what was decided, what's next.

Memory fills with outdated decisions, redundant orientation facts, and content that belongs in Preferences or knowledge files. Claude's per-conversation orientation degrades as Memory becomes noisy.

Audits for redundancy and staleness, identifies content ready for promotion or demotion, and produces specific edit prescriptions with human-in-the-loop confirmation.

Custom Instructions

The behavioral architecture of your Project. Identity, rules, routing, output standards. Everything here loads into every conversation, every time. This is the highest-leverage layer: the right instructions here shape every response Claude produces.

Instructions that work individually conflict with each other silently. Rules meant for knowledge files bloat the always-loaded context. Behavioral calibration is guesswork. Seven structural anti-patterns hide in plain sight.

Builds Custom Instructions using the 5-layer prompt architecture. Scores instruction clarity across six dimensions. Detects anti-patterns and provides tested countermeasures for Claude's eight behavioral tendencies.

Knowledge Files

Searchable reference depth. Documentation, data, and frameworks loaded on demand when a query matches their content. In retrieval mode, Claude searches rather than loading everything. How files are structured directly affects what gets found.

Files structured for human reading fail Claude's retrieval patterns. Orphaned files go unreferenced. Context budget gets consumed by content that belongs in other layers.

Runs full budget analysis with per-file evaluation, content routing, growth trajectory, and retrieval quality audit. Detects orphaned files and content misplacement.

Conversation Context

The current exchange. Where you interact with the Project. Two Skill categories activate here in the background — block libraries and domain packs — contributing structure, identity, reasoning frames, output formats, and domain expertise as needed.

If the layers above are misconfigured, conversation quality suffers in ways that look like Claude's fault but are actually structural. Output drift, ignored instructions, and degraded quality over long conversations are symptoms of architectural problems.

Skills activate through this layer automatically. Keywords and task descriptions trigger the right diagnostic and compilation tools without requiring explicit commands.

Cross-Layer

The only Skill that operates across multiple scopes simultaneously. Detects misalignment between Global setup and Project setup that single-scope audits cannot see.

Single-scope audits cannot see cross-scope drift. A rule in Custom Instructions might fight a User Preference. A Knowledge File might duplicate something already in Global Memory.

rootnode-full-stack-audit runs Project audit + Global audit + Cross-Layer Alignment in a single pass — the only Skill in the catalog that operates across scopes rather than within one. Detects collisions across scopes and produces a unified action plan ordered by impact.

Claude Code · Global User Settings

The ~/.claude/ directory containing personal settings, MCP servers, and user-installed Skills available across all repos. Where dual-surface Skills live on the Claude Code side.

Drift between user-level settings and repo-level expectations leads to inconsistent execution across repos.

cc-design installs at the user level — available to act on any Claude Code repo without per-repo reinstallation.

Claude Code · Repo

The repo's .claude/ directory containing repo-specific rules, agent definitions, hooks, and policies. The execution layer — where autonomous work happens against the codebase. Reads configuration profiles produced upstream by profile-builder.

Claude Code repos drift over time — patterns calcify, anti-patterns accumulate. Risky changes need a gate before they hit the repo, not after.

critic-gate provides pre-commit safety review. mode-router routes work by profile-driven strictness per file class. repo-hygiene runs a 14-category sweep plus 7-layer leak check across an existing Claude Code deployment.
