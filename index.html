01 — The problem

How well is your Claude Project actually performing?

A Claude Project that produces decent output can still be structurally unsound. There's no scorecard. No structural diagnostics. No built-in way to know whether your Project is working at 60% of its potential or 95%. root.node gives you the diagnostic tools to find out and the compilation pipeline to fix what you find.

02 — The architecture

Nine layers. One system.

Every Claude Project sits within a nine-layer architecture. Five global layers form the foundation that spans all your Projects. Three project layers shape behavior within a single Project. The conversation layer is where you interact with the system.

How content is distributed across these layers determines everything. A rule in the wrong layer gets ignored. A preference that should be global gets restated in every Project. Context budget gets consumed by content that belongs somewhere else. Cross-layer conflicts cause unpredictable output that no amount of prompt rewording will fix.

root.node treats the entire architecture as a unified system: scoring each layer, detecting cross-layer conflicts, diagnosing Claude's behavioral tendencies, and building Projects that are structurally sound from the start. The system is calibrated to how Claude actually processes context: the behavioral tendencies documented through extensive testing, the loading behavior of each layer, and the architectural patterns that produce reliable output.

04 — In practice

What it looks like to use root.node.

Install a Skill. Talk to Claude about your Project. The right tools activate automatically.

Audit an existing Project

You say

"Audit my project"

You get

A scored diagnostic with the specific text causing each issue, the pattern it matches, and tested fixes ranked by impact. Act on the findings in the same conversation.

05 — The workflow

Compile, don't compose.

Traditional Project design is composition: write instructions, test them, tweak by feel, hope they hold. root.node replaces that with compilation: a structured process that assembles complete Claude Project architectures from tested components.

The Compiler

Build Projects from tested components

1

Parse

Analyze the task. What does the Project need to accomplish? What domain expertise, reasoning depth, output format, and behavioral calibration does it require? In Project Mode, the Compiler also reads your global configuration (User Preferences, installed Skills, and MCP Connectors) to inform the scaffold.

2

Select

Choose the right blocks. Decision trees map task characteristics to the best identity approach, reasoning variant, and output format from the tested library. Domain packs activate automatically for specialized tasks.

3

Construct

Assemble the Project using the 5-layer prompt architecture (Identity, Objective, Context, Reasoning, Output + Quality Control). Blocks slot into their layer. Knowledge file architecture, Memory design, and a global layer advisory are included in the scaffold.

4

Validate

Score the compiled Project against 6 quality dimensions. Check global layer alignment. Catch architectural flaws before deployment, not after.

The Optimizer

Diagnose and fix existing Projects

1

Parse

Ingest the existing Project architecture: Custom Instructions, knowledge files, Memory. For global and full-stack audits, also parse User Preferences, Styles, installed Skills, and configured MCP Connectors.

2

Diagnose

Score against the 6-dimension Project Scorecard and, for global audits, the 6-dimension Global Layer Scorecard. Detect structural anti-patterns, behavioral miscalibrations, and cross-layer failure modes, all with evidence from the Project materials.

3

Prescribe

Produce targeted fixes ranked by impact. Every prescription references the specific text causing the issue and the tested countermeasure. For full-stack audits, produce evolutionary recommendations: promotions, demotions, codifications, and Skill extractions.

4

Reconstruct

Optionally rebuild the Project architecture from the ground up using the Compiler methodology, informed by everything the diagnosis revealed.

06 — The audience

For practitioners who build with Claude.

root.node is built for people who use Claude Projects for real work. Not casual prompt experimentation. Structured, repeated workflows where output quality directly affects outcomes.

You'll get the most from these Skills if you're already building Projects and want to understand what's working, what's not, and why. Whether that means auditing a Project you've been running for months, scaffolding a new one from tested components, or diagnosing why quality degrades over long conversations.

User Preferences

Communication style, expertise level, behavioral defaults. Everything here persists across all Projects and all conversations. The foundation layer: what Claude knows about you before any Project loads.

Preferences are too vague to guide Claude's behavior. Redundant rules get restated in every Project. Stable patterns stay trapped in Project Memory instead of being promoted globally.

Evaluates Preference precision, detects redundancies with Project-scoped instructions, and identifies patterns ready for promotion from Project Memory.

Styles

Tone, format, and delivery patterns. Applied by selection and can be switched per conversation. Controls how Claude communicates, separate from what it knows or how it reasons.

Style settings conflict with formatting rules hardcoded in Custom Instructions. Tone preferences get scattered across Projects instead of centralized.

Detects tension between Style settings and Custom Instructions formatting. Identifies when delivery preferences belong in a Style rather than being hardcoded per Project.

Global Memory

Cross-project context and learned preferences. Auto-synthesized from conversations. Orientation facts that help Claude understand who you are and what matters to you, loaded into every conversation.

Memory accumulates stale facts, redundant entries, and reference-depth content that belongs in knowledge files. Bloated Memory consumes context budget across every conversation.

Audits for redundancy and staleness. Rebalances content between Memory, Preferences, and knowledge files. Tracks Memory's contribution to context overhead.

Skills

Procedural knowledge packaged in SKILL.md files. Activated automatically by task relevance. Skills use progressive disclosure: a lightweight description loads at startup, the full methodology loads on demand.

Installed Skills collide with Project knowledge files covering the same territory. The Skill portfolio has gaps that force manual workarounds. Overhead from unused Skills consumes context budget.

Evaluates Skill portfolio fitness, detects collisions with knowledge files, and converts design specs into deployment-ready Skill packages.

MCP Connectors

External service integrations like Gmail, Slack, Google Drive, and Notion. Tools loaded on demand through deferred loading: configured once, available everywhere.

Projects assume connector capabilities that aren't configured. Connector overhead goes untracked. Alignment between available connectors and Project needs drifts as Projects evolve.

Checks connector alignment with Project needs, detects assumed capabilities without configured connectors, and accounts for connector overhead.

Project Memory

Phase awareness, active constraints, key decisions. Auto-synthesized per Project. The orientation layer that helps Claude pick up where you left off: what you're working on, what was decided, what's next.

Memory fills with outdated decisions, redundant orientation facts, and content that belongs in Preferences or knowledge files. Claude's per-conversation orientation degrades as Memory becomes noisy.

Audits for redundancy and staleness, identifies content ready for promotion or demotion, and produces specific edit prescriptions with human-in-the-loop confirmation.

Custom Instructions

The behavioral architecture of your Project. Identity, rules, routing, output standards. Everything here loads into every conversation, every time. This is the highest-leverage layer: the right instructions here shape every response Claude produces.

Instructions that work individually conflict with each other silently. Rules meant for knowledge files bloat the always-loaded context. Behavioral calibration is guesswork. Seven structural anti-patterns hide in plain sight.

Builds Custom Instructions using the 5-layer prompt architecture. Scores instruction clarity across six dimensions. Detects anti-patterns and provides tested countermeasures for Claude's eight behavioral tendencies.

Knowledge Files

Searchable reference depth. Documentation, data, and frameworks loaded on demand when a query matches their content. In retrieval mode, Claude searches rather than loading everything. How files are structured directly affects what gets found.

Files structured for human reading fail Claude's retrieval patterns. Orphaned files go unreferenced. Context budget gets consumed by content that belongs in other layers.

Runs full budget analysis with per-file evaluation, content routing, growth trajectory, and retrieval quality audit. Detects orphaned files and content misplacement.

Conversation Context

Messages, uploads, tool results. Where architecture meets execution. This is the layer where you interact with the Project and where Skills activate through natural language.

If the layers above are misconfigured, conversation quality suffers in ways that look like Claude's fault but are actually structural.

Skills activate through this layer automatically. Keywords and task descriptions trigger the right diagnostic and compilation tools without requiring explicit commands.
